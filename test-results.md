# LLaMA3 단일 구동 대비 Gemma:2b 하이브리드 라우팅 성능 비교 지표
# Local LLM 하이브리드 라우팅 성능 벤치마크 리포트

본 문서는 제한된 가상 인프라 자원 환경(CPU-only / 가상화 리눅스 서버)에서 단일 대형 모델(LLaMA3) 운용 대비 초경량 모델(Gemma:2b)을 활용한 동적 라우팅 아키텍처의 자원 효율성 및 성능 비교 측정 데이터입니다.

## 1. 테스트 환경 (Infrastructure Specification)

* **OS:** Enterprise Linux (Kernel 5.x)
* **CPU:** vCPU 4 Cores
* **Memory:** 16GB RAM (No Hardware GPU Acceleration)
* **LLM Engine:** Ollama Backend Engine

## 2. 모델별 자원 점유율 및 지연 시간 비교 표
| 평가지표 | 시나리오 A: LLaMA3 단일 RAG 운용 | 시나리오 B: Gemma:2b 하이브리드 라우팅 (PoC) | 개선율 (Efficiency) |
| :--- | :--- | :--- | :--- |
| **초기 모델 로드 시간** | 약 45초 ~ 1분 | **약 5초 내외** | **80% 이상 단축** |
| **일반 질의 레이턴시** | 평균 15.4초 (RAG 연산 강제) | **평균 1.2초 (RAG 바이패스)** | **92.2% 속도 향상** |
| **보안 질의 레이턴시** | 평균 18.2초 (추론 과부하) | **평균 3.5초 (경량 RAG 격발)** | **80.7% 속도 향상** |

| **메모리(VRAM) 할당량** | 약 4.8 GB | **약 1.6 GB** | **메모리 락 우려 해소** |

## 3. 엔지니어링 분석 결과 (Trade-off & Insight)

### [장점: 인프라 가용성 극대화]
* `Gemma:2b` 라우터 체인을 구축함으로써, 시스템의 CPU 전수 독점 현상을 방지하고 멀티스레드 환경에서의 시스템 데몬 가용성을 확보함.
* 7월 양산형 관제 자동화 시스템으로 안착시키기에 충분한 응답 속도 지표를 확보함.

### [한계점 및 향후 고도화 과제]
* **컨텍스트 드랍:** 모델 경량화 과정에서 한국어 문맥 해석력이 다소 저하되어, RAG 연산 결과 출력 시 일부 영문 믹싱 현상이 관측됨.
* **대응책:** 1차 라우터는 속도가 빠른 `Gemma:2b`로 상시 가동하되, 중요 보안 지침을 생성하는 백엔드 코어에는 `LLaMA3-Ko` 또는 중간 스케일의 모델을 하이브리드로 교차 검증(Cross-Validation)하는 멀티 큐 파이프라인으로 2단계 고도화 예정.
