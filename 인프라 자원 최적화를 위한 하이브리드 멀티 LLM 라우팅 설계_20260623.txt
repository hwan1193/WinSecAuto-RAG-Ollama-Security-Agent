인프라 자원 최적화를 위한 하이브리드 멀티 LLM 라우팅(Routing) 설계

단일 대형 모델 ('LLaMA3') 기반 RAG 파이프라인의 높은 CPU/메모리 점유율과 응답 지연 (Latency) 문제를 해결하기 위해,
사용자의 질의 의도를 파악하여 트래픽을 동적으로 분기하는 "LangChain LCEL 기반 하이브리드 멀티 LLM 아키텍처"를 추가 구축하고 검증했습니다.

### 1) 멀티 LLM 분기 아키텍처 개념

* 1차 라우터 레벨: 초경량 모델인 'Gemma:2b'가 사용자의 입력을 실시간으로 컨텍스트 분석하여 `SECURITY`(보안 규정 질의)와 `GENERAL`(일반 대화)로 분류합니다.

* 2차 실행/추론 레벨: `SECURITY` 분류 시: 로컬 `FAISS` 벡터 데이터베이스 및 메인 RAG 파이프라인을 기동하여 정밀 보안 지침을 탐색합니다.

* `GENERAL` 분류 시: RAG 연산과 무거운 컨텍스트 조회를 완전히 바이패스(Bypass)하고, `Gemma:2b`가 인라인으로 즉시 응답 패킷을 생성하여 시스템 자원을 보존합니다.

### 2) 고도화된 동적 라우팅 소스코드 ('Guidelines.py')

==========================================================================

```python
from langchain_community.llms import Ollama
from langchain_community.document_loaders import TextLoader
from langchain_community.vectorstores import FAISS
from langchain_huggingface import HuggingFaceEmbeddings
from langchain.chains import RetrievalQA
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 1. 멀티 LLM 추론 엔진 바인딩 (메인 RAG 엔진을 Gemma로 전면 스위칭)
# [변경 완료] 메인 RAG 추론 엔진을 무거운 LLaMA3에서 초경량 Gemma:2b로 교체
main_llm = Ollama(model="gemma:2b")
# 1차 라우팅용 초경량 모델
router_llm = Ollama(model="gemma:2b")

# 2. 로컬 임베딩 및 문서 로드
embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
loader = TextLoader("./Guidelines.txt", encoding="utf-8")
documents = loader.load()
vector_store = FAISS.from_documents(documents, embeddings)

# 3. 메인 RAG 체인 구성
qa_chain = RetrievalQA.from_chain_type(
    llm=main_llm,
    chain_type="stuff",
    retriever=vector_store.as_retriever()
)

# 4. 1차 라우터 프롬프트 (무결성 유지)
router_prompt = PromptTemplate.from_template(
    "<|begin_of_text|><|start_header_id|>system<|end_header_id|>\n"
    "You are a routing machine. You must reply with exactly one word: either 'SECURITY' or 'GENERAL'.\n"
    "Never write any other text, explanation, or punctuation. Just one word.\n"
    "If the question is about security, guidelines, password, policy, rule -> SECURITY\n"
    "If the question is greetings, weather, small talk, general info -> GENERAL\n"
    "<|eot_id|><|start_header_id|>user<|end_header_id|>\n"
    "Question: {question}\n"
    "<|eot_id|><|start_header_id|>assistant<|end_header_id|>\n"
)
router_chain = router_prompt | router_llm | StrOutputParser()

# 5. 동적 라우팅 함수 정의
def route_and_execute(user_question):
    raw_decision = router_chain.invoke({"question": user_question}).strip()

    if "SECURITY" in raw_decision.upper():
        decision = "SECURITY"
    else:
        decision = "GENERAL"

    print(f"\n[⚙️ 인프라 라우터 로깅] 분류 결과: {decision} (원문: {raw_decision})")

    if decision == "SECURITY":
        # [로그 변경] 메인 RAG 모델 변경에 따른 출력 가시성 확보
        print("[🔒 보안 관제 라인] 중요 보안 지침 탐지 - Gemma:2b 메인 RAG 파이프라인 가동.")
        return qa_chain.run(user_question)
    else:
        print("[💬 일반 통신 라인] 단순 질의 탐지 - Gemma:2b 인라인 즉시 응답.")
        return router_llm.invoke(user_question)

# 6. 인프라 모의 격발 테스트
if __name__ == "__main__":
    q1 = "우리 회사 보안 규정 중 패스워드 변경 주기는 어떻게 돼?"
    res1 = route_and_execute(q1)
    print(f"결과 리포트 1:\n{res1}\n")
    print("-" * 50)

    q2 = "안녕 반가워! 오늘 날씨가 참 좋네."
    res2 = route_and_execute(q2)
    print(f"결과 리포트 2:\n{res2}\n")

4) 엔지니어링 인사이트 및 분석 결과 (Engineering Summary)

1. 가용성 및 응답성 개선 : 기존 LLaMA3 단일 운용 대비 VRAM 및 CPU 자원 병목 현상이 완벽히 해소되었고 시스템 응답 속도가 비약적으로 향상됨을 확인함.

2. 트레이드오프 (Trade-off) 관측 : 초경량 모델 (Gemma:2b) 특성상 RAG 답변 생성 시 일부 영문 컨텍스트가 믹싱되거나 한국어 문맥 유지력이 대형 모델에 비해 미흡한 현상을 확인함.

3. 결론 : 실안착 인프라 구성 시 라우터 엔진은 초경량 모델 (Gemma)로 고정하여 속도를 확보하되, 고도의 보안 지침 추론이 필요한 RAG  백엔드에는 프롬프트 튜닝을 고도화하거나 파라미터가 조금 더 큰 모델을 하이브리드로 조합하는 것이 최적의 아키텍처 선별 기준임을 실증함.
